curryGPU ISS
功能级模拟器

在写任何 RTL 之前，用一台可执行的模型确证 ISA 的功能完备与自洽：能编码、能解码、能执行、能跑通真实 workload 并产出 bit-exact 正确结果。它既是 ISA 的可执行规范、又是后端 RTL / 编译器的正确性 oracle。无时序——只演化架构状态，不模拟周期与管线。北极星 = 端到端跑通一个小 Transformer block。

00

定位：可执行的规范

spec · oracle · bench

硬件工程里有一个反复出现的教训：先有可执行的真值，再写 RTL。CPU 世界里这台真值叫 Spike（riscv-isa-sim）；GPU 世界没有现成的功能级先例，于是 curryGPU 自建一台。它先于硅、先于 RTL 存在，回答一个问题：「这套 ISA 真的能算对吗？」

三重身份

身份含义谁来用
可执行规范
executable spec		ISA 的「意义」由 ISS 执行后的架构状态定义——文字规范有歧义时，以 ISS 跑出来的结果为准ISA 设计者 · 编译器后端
正确性 oracleRTL / 编译器产物与 ISS 逐位 diff；任何不一致都是被测物的 bug，不是 oracle 的RTL DV · 端到端验证
workload 运行台真实 kernel（attention / FFN / LayerNorm）在硅就绪前先在这里跑通、调对kernel 开发 · 数值研究

功能 ≠ 时序：最大的速度杠杆

ISS 只建功能核：一条指令进来，架构状态（寄存器 / 谓词 / 内存 / PC）原子地跳到下一态。没有周期、没有管线、没有 warp 调度策略、没有 cache——这些是 timing 模型的关注点，与「算得对不对」无关。把它们剥离，本身就是最大的速度杠杆（纯功能模式较 timing 模式快约 5–10×，源 GPGPU-Sim）。功能与时序严格分离，但留干净的回调边界，供日后另起一层包 timing。

一条贯穿全文的红线功能 oracle 只需要「对尊重同步契约的程序，最终架构态唯一确定」这一条性质。它不关心指令以什么顺序、什么速度到达——只要终态可复现、可对拍。后面 §05 会看到，这条红线如何把 GPU 里最棘手的「线程分歧」收敛成一个可证、可测的设计。

01

SIMT 翻转

why GPU ISS ≠ CPU ISS

要理解 ISS 的每个设计决策，先理解一件事：GPU 功能 ISS 不是「并行版的 CPU ISS」，它把复杂度的位置整个翻转了。这一节是全文的概念地基。

串行 CPU ISS（Spike）持一份标量状态、核心循环逐条推进——复杂度全在指令语义（一条 FFMA 怎么算）。SIMT 把图景反转：单条指令的标量语义平凡，复杂度搬到了「一条指令跨 32 个 lane 复制执行，且要决定哪些 lane 此刻活跃」。

五个结构性差异（每一个都对应一个设计决策）

#CPU（Spike）SIMT（curryGPU ISS）带来的决策
1执行单元 = 标量32-lane 向量：warp 是 fetch/decode 单位，lane 是架构状态单位；每指令带 active maskdecode-once + masked-lane loop（§04）
2XPR[32][warp×lane×reg] 三维，且分 scalar(uniform) / vectorSoA vgpr[regID][32]（§04）
3改写一个 PCnext-PC-per-lane + lane 如何重新分组——Spike 无对应物ITS per-PC grouping（§05）
4无warp/CTA/cluster 真同步层级；但调度策略是 timing 关注点功能 oracle 取一个确定序即可（§05/§08）
5单一扁平内存按 space 分离实例化（global/shared/local/const + 张量近存）memory_space per-space（§07）
教学要点记住这张表：后面每一节都是在回答「翻转之后，这件事该怎么做」。第 3 行（分歧）是 GPU ISS 独有、且最难的部分，§05 整节都在讲它。
02

系统架构

single source → core → coarse boundary

一条主数据流：一份声明式编码表同时生成 C++ 解码器与 Python 汇编器；kernel 经汇编器变成 128-bit 指令流，喂进 C++ 功能核；功能核与 Python 只在粗粒度边界（launch / step / state-diff）交互；测试框架在 Python 侧做差分与 conformance。

单一源 / 代码生成C++ 功能核 Python 测试 / 工具pybind 粗边界

四条结构原则

  • 单一真相源、双向：编码表既生成解码器、又生成汇编器，二者必然一致（round-trip 可证，§03）。绝不手维护第二份编码。
  • 功能 / 时序严格分离：现在只建功能核，留干净回调边界供日后包 timing；不让性能关注点污染正确性核。
  • 粗粒度 pybind 边界：跨语言调用次数 = O(launch + step)，与指令条数无关——step(max_steps) 在 C++ 内一直跑到 EXIT / 陷阱 / 预算，不每条指令回 Python。
  • 依赖单向 isa → iss：iss build 时调 isa 生成器，严禁反向，杜绝循环依赖（§09）。
03

单一源 → 代码生成

declarative table · bidirectional

编码表是权威源，直接驱动 C++ 解码器、Python emit() 汇编器与 JSON IR。生成时做两道门：完备性（每条指令都有合法绑定、每个字段宽度/偏移/符号扩展都对）与无重叠（没有两条指令的位 cube 相交）。这是「编码正确」的强保证。

schema 公开 / layout 私有：引擎与编码无关

编码表拆两层。schema（公开）声明指令、操作数、语义、字段名与宽度；layout（私有）声明 bit 偏移、opcode 数值、fsel 映射。生成器 = schema + layout → 解码器 / emit / IR。关键纪律：语义 handler 只消费已解码的字段，从不触碰 raw bit——于是「引擎」与「精确编码」解耦，工程开源而生产编码可私有（公库带样例 layout 够 build / CI / demo，私有 submodule 注入生产 layout）。

128-bit 定长是个工程难点：主流工具（QEMU decodetree、LLVM FixedLenDecoderEmitter）对 >32-bit 支持弱，需要 APInt 式宽字段提取，所以生成器核心是自写的——按 opcode / modifier 判别位构造 bit 区分树。.（匹配位）与 -（忽略位）区分 reserved 与控制段；MMA / TMA / sparsity / 寄存器对 / 控制段等用手写 C++ 语义钩子接手已解码字段。

04

执行核

decode-once · masked-lane · SoA

功能核的两个核心动作：decode-once（一次解码服务一个 PC 上所有 lane）与 masked-lane loop（循环 32 lane，active=0 的 lane 跳过、不写寄存器或内存）。寄存器堆用 SoA（structure-of-arrays）布局，为日后向量化铺路。

warp 状态：数据结构蓝图

// 一个 warp 的全部架构状态（功能核私有成员）
warp_state {
  vgpr   : uint32[NUM_VREG][32]   // SoA · 无类型 32-bit word（类型随指令 modifier）
  ureg   : uint32[NUM_UREG]       // per-warp uniform 标量寄存器
  pred   : bitmask Pn[NUM_PRED]    // per-warp 谓词（1 bit / lane）
  pc     : uint32[32]            // ★ ITS：per-thread PC（§05 的关键）
  lane_state : {active,blocked,yielded,exited}[32]
  Bx     : {participation_mask, reconv_pc, valid}[16]  // 收敛屏障（§05）
}
无类型 word
GPR = 无类型 32-bit word（编码里的「8-bit」是寄存器索引、非数据宽度）；类型随指令 modifier 解释。
sub-word
8/16-bit 结果 extend-to-32（按签名 sign/zero）——循 ARM AArch64 / RISC-V，无 partial-register merge，拒 x86 AL/AX 合并 hazard。
宽寄存器
64-bit = 偶对齐对 R(2n):R(2n+1)；128-bit = 4-对齐组；packed = SIMD-within-word（小端 lane）。
阶段 ① 已就位上面 pc[32] / lane_state[32] / Bx[16] 在地基阶段就已建好——但当前是惰性的：调度器假定所有 active lane 共享一个 PC，一旦分歧就 trap。把它们激活，就是阶段 ② ITS（§05）。
05

ITS：分歧与重聚

per-PC grouping · convergence barriers

这是 GPU 功能 ISS 独有、且最难的部分，也是本文的核心。问题很具体：当 warp 内 32 个 lane 走到不同的 PC（分歧），一台没有「重聚栈」的功能 oracle 怎么正确地执行、并在该重聚时重聚？

active laneblocked（待重聚） 选中执行的组exited / 非法

核心机制：per-PC grouping

不用经典的 IPDOM 重聚栈（GPGPU-Sim / Vortex / Ventus 都是栈式）。curryGPU 用 per-PC grouping：每一步——

  • ① 把 active lane 按相同 pc[lane] 分组；
  • ② 按一个确定的全序（min-PC-first 为默认）选一组，decode 一次，只对该组 mask 执行；
  • ③ handler 算 per-thread next-PC，lane 自然分裂 / 推进。warp 退休 = 全 lane exited。

这不是简化，而是 post-Volta ITS 的架构语义本身：同 PC 的 warp 内线程可在任意周期同调，重聚可早 / 晚 / 完全省略，唯一约束是 warp 内显式同步（Hanoi arXiv:2407.02944 §IV；NVIDIA Volta whitepaper；Simty 是其硬件存在性证明）。功能 oracle 取最廉价的合规设计即可。

收敛屏障：分歧/重聚怎么被记录

per-thread PC 解决「lane 能在不同 PC」，但「该在哪重聚」需要显式状态——16 槽收敛屏障 Bx{participation_mask, reconv_pc, valid}，由六条控制指令驱动。它不是单 IPDOM 栈，而是独立的 barrier 对象（NVIDIA 收敛屏障专利 US10067768B2 的功能等价）。

指令语义（功能核）对状态的效果
BSSY Bx,t在分歧区之前建立屏障：快照当前 PC-group 为参与集，记重聚目标 tBx={participation=GROUP_mask, reconv_pc=t, valid=1}
BSYNC Bx到达 lane 进入 blocked；当参与集全部 ∈ {blocked, exited} 时，实际到达的存活子集在 reconv_pc 重激活fire 谓词 participation ⊆ (blocked|exited)　（yielded 不计入）
BREAK Bx把 lane 从参与集清位（loop break / 早退；防 BSYNC 永等）participation &= ~bits(M)
CONT分支回 loop header（非 barrier op；重聚由该 loop 自己的 Bx 管）—（BRA 回边）
YIELD纯调度提示、零架构效果：促调度器下步选另一 PC-group（让自旋 / 生产者-消费者推进）lane→yielded（仍 runnable）
EXIT/KILL/RETlane 退休，并立即从所有 valid Bx 清位（starvation-free 硬前提）lane→exited，∀Bx 清位

一个 if/else diamond 怎么跑

编译器把分歧区 lower 成「BSSY 立屏障 → predicated BRA 分流 → 两路各自执行 → BSYNC 在 join 重聚」。分歧由 BRA 的谓词承载，BSSY/BSYNC 本身无谓词。

L0: BSSY  B0, L_join     // participation = {全 32 lane}, reconv = L_join
L1: @P0 BRA L_else     // P0 真→落 L2(then)，P0 假→跳 L_else
L2: IADD3 R4,R4,R1    // then 块：只有 P0-true lane 在此组
L3: BRA  L_join
L_else: IADD3 R4,R4,R2 // else 块：P0-false lane
L_join: BSYNC B0          // 两路到齐 → 32 lane 在此重聚，B0.valid=0
       EXIT

▍ active mask 随这段代码的演化（黄=active，蓝=本步选中执行的组，粉=blocked 待重聚）

01234567··31 L0 BSSY：32 lane 全 active，全部登记进 B0.participation
01234567··31 L2 then：min-PC 选中 then 组（蓝）执行；else lane 在 L_else 挂起
01234567··31 L_join BSYNC：先到的 lane blocked（粉），待全 participation 到齐 → fire
01234567··31 fire：32 lane 在 reconv_pc 重激活，重新收敛

lane 状态机

核心性质：最终态与调度序无关

这是功能 oracle 唯一真正需要、也是阶段 ② 主验收门的性质：对尊重同步契约的程序，最终架构态与「组选择序」无关。为什么成立？因为——每 lane 的架构态私有；所有跨-lane 交互都经显式重聚或 membermask ⊆ active 门控；而 warp 内无屏障隔离的内存竞争本身就是 UB（无需序无关）。于是不同的合规调度序只是「同一组私有状态机的不同交错」，终态唯一（闭包性质，参见 Habermaier-Knapp ESOP 2012）。

已锁定的设计决策（来自 ITS 调研）participation = GROUP-mask（跟 NVIDIA 对齐，非 guard 子集）；屏障效果只施于 guard-true lane，BSSY/BSYNC 无谓词（predicated 形式判为 out-of-scope，避死锁），BREAK/YIELD 可谓词；collectives（ELECT+VOTE）以最小 NV 集入阶段 ②。完整推导与对抗评审见 docs/implement/ISS/research-its.md。
06

张量与 bit-exact

leader-issue MMA · FDA(F=25)

MMA 不模拟脉动阵列的 dataflow——功能上一次算完整 bit-exact tiled matmul。难点不在「怎么乘」，而在「怎么累加才能逐位可复现」：张量核的累加顺序、对齐、舍入决定了最后一个 bit。

leader-issue 宏-op

ELECT 在 membermask 内选唯一 leader lane（最低 active lane）发描述符，非 leader lane 在该 PC 跳过；ITS 收敛屏障保证 warp 在 MMA 处收敛、同步前进。A/B 经 shared descriptor（驻 UR，载 shape/stride/tile/swizzle）取数；C/D 累加器驻张量近存（tensor-mem），不进 GPR fragment。这是单指令、warp 级的宏操作，对齐 Blackwell tcgen05。

统一 FDA(F=25)：累加规范

单条 MMA（一个 K-tile）算 d = c + Σ a_k·b_k，五步——所有 .kind（F16/BF16/TF32/FP8/FP6/FP4/MX/NVFP4）统一，不抄各架构的 per-format 算法动物园（那正是跨架构数值不一致的来源）。

F=25
所有 .kind 统一；≥ FP32 的 24-bit significand，论文建议「≥23 bit，越多越好」。
全 FP32 范围
输入/乘积/累加器/输出皆不 FTZ、保留 subnormal；定点累加器无中间溢出。
仅对称舍入
对齐截断 RZ（fused 固有）+ 末次 RNE；不用 RD，避开 round-down 的系统性负偏。
SFU
MUFU.EX2/LG2 取 correctly-rounded（RN(2^x)，≤0.5 ULP vs MPFR）；exp = exp2(x·log2e)。
07

内存空间

per-space · sparse · gather/scatter

内存不是单一扁平数组，而是按 space 分离实例化：抽象基类 memory_space + 稀疏实现 memory_space_impl（4KB 块 hashmap），每个空间一个实例。32-lane 访存包成 gather/scatter，支持 byte-masked 写。

地址空间集与 ISA 对齐（见 ISA §02）：{register, tensor-mem, shared/L1, const, local, global}。cluster / DSMEM 延后，只留非破坏接缝——cluster_dim 默认 1（= 恒等，单-CTA 行为 bit 不变）；真·跨-CTA 访问干净 unsupported trap，绝不伪造远端态。

09

工程落位与边界

isa/ + iss/ · one-way dep

仓库根 = 整个 curryGPU 工程（非仅 ISS）。顶层 isa/ 是 ISA 单一源（被 ISS 现用、未来 RTL / 汇编器 / 文档共享）；iss/ 是功能 ISS 独立子树。构建按组件自治，依赖严格单向。

curryGPU/
├── docs/                  # design/(isa.html · iss.html · index) + implement/ISS/
├── isa/                   # ISA 单一源（顶层共享）
│   ├── schema.py          # 声明式编码表（公开 · 真源）
│   ├── layout/            # bit 偏移 / opcode（样例公开 · 生产私有 submodule）
│   ├── codegen.py         # 表 → C++ 解码器 + IR + 完备/无重叠门
│   └── assembler.py       # 表驱动 emit()（双向）
├── iss/                   # 功能 ISS（独立子树 = 当前构建目标）
│   ├── CMakeLists.txt      # codegen(DEPENDS ../isa) → C++ 核 → pybind → ctest
│   ├── binding/native.cpp  # C++ 功能核 + 粗 pybind（launch/step/state-diff）
│   └── build/ (gitignored) # 生成的 decode.gen.cpp / isa_ir.json
└── (rtl/ dv/ sw/ — 未来按需，现在不建空目录)
  • 依赖单向 isa → iss：iss build 时调 isa 生成器，严禁反向（防循环）。
  • 粗 pybind 边界：仅 launch / step(max_steps) / state-diff；跨边界调用数 O(launch+step)，与指令数无关。
  • 生成物入 iss/build/、不入库：CI 用「两次构建逐字节一致」+ git ls-files 私有化门替代 git-diff 式 regen-clean。
  • currygpu 作 namespace package 横跨 isa/（纯 Python）+ iss/（C++ 扩展）；仓库根不放 Python 包文件（根是 HW 工程根）。
